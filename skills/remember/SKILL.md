# Remember

在你系統提示的「# auto memory」段落裡，已經定義好記憶系統的位置、格式（frontmatter: name/description/metadata.type）、四種類型（user/feedback/project/reference）、寫入的兩步驟（寫檔案 + 更新 MEMORY.md 索引）、以及「不要存什麼」的排除清單。這個 skill 不重複那些規則，只負責「觸發時機」與「互動流程」，實際寫入時照你系統提示裡那套規則做。

如果目前 session 的系統提示裡沒有「# auto memory」段落（代表這個專案還沒有啟用自動記憶系統），告知使用者並停止，不要自己另建一套格式。

## 觸發

**手動**：使用者直接說要記住/存記憶（「記住這個」「幫我存起來」「/remember」等）。

**主動**：對話中只要出現一個新的、經過釐清的結論／決定／要做的事，就主動問一次要不要存。不要等到整個話題結束才問——每個決議都問，但只問一次、一句話帶過，不要打斷正在進行的討論節奏太久。

## 流程

1. **主動觸發時，先問再做**：只問「這個要不要記起來？」這一句，等使用者回答。使用者說不要就跳過，不要草擬任何內容。
2. **草擬**：根據剛才的討論，比對系統提示裡的四種類型定義與排除清單，決定：
   - 是否真的值得存（排除清單裡的東西一律跳過，並直接告知使用者「這個不用存，屬於 XX」，不用問）
   - 屬於哪個 type
   - name（kebab-case）、description、內容（feedback/project 類型要有 **Why:** 和 **How to apply:** 兩行）
   - **topic 候選**：想 2-3 個未來最可能用來聊到這件事的主題關鍵字/短語（例如「skill 開發」「記憶系統設計」），這是給 MEMORY.md 分組與未來觸發回憶用的，跟四種 type 是不同維度的分類
   - 檢查 MEMORY.md 索引有沒有可以更新的既有相關記憶，或是有沒有語意相近的既有 topic 分組可以歸進去，而不是硬存一則新的重複記憶、或另開一個跟既有分組意思重疊的新 topic
3. **給使用者看草稿**，內容連同 topic 候選一起用簡短文字或條列呈現，讓使用者一次確認/修改內容、並從 topic 候選中選一個（或自己指定）。不要用逐項提問去引導使用者自己寫——你先寫好，使用者改比較快。
4. **確認後才寫入**：
   - 記憶檔案本身照系統提示「auto memory」段落的 frontmatter 格式寫（name/description/metadata.type），另外多加一個 `topic: <選定的主題>` 欄位——這是這個 skill 額外加的，不在基礎系統定義內，但只是新增欄位不影響基礎系統讀取。
   - MEMORY.md 索引依 topic 分組呈現，用 `## <topic>` 當標題分節，同一 topic 的記憶條目放在同一節下面；選定的 topic 若跟既有標題語意相同就併進去，不要開重複的節。

## 什麼時候不用問、不用存

系統提示「auto memory」段落裡「What NOT to save in memory」的排除清單優先於這個 skill 的一切判斷——程式碼慣例、git 歷史、debug 過程、CLAUDE.md 已經寫的東西、暫時性的當次任務細節，都不問、不存。
